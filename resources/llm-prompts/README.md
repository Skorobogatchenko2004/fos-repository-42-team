# LLM-сервисы и промпты

Раздел содержит промпты, применяемые в оценочных средствах дисциплины. Промпт непереносим между моделями без перепроверки. При регистрации указываются модель, назначение, входные данные, ожидаемый формат результата и способ верификации.

| Промпт | Применяет | Назначение | Связанные КИМ |
|---|---|---|---|
| Критик требований | обучающийся | самопроверка спецификации перед сдачей | КИМ-2.5 |
| Формальный судья | преподаватель (CI) | формальная проверка полноты документа | все КИМ |
| Генератор дефектного артефакта | преподаватель | подготовка материала для инспекции | КИМ-2.5, КИМ-3.4 |

## Критик требований

Применяется обучающимся для самопроверки. Промпт не исправляет спецификацию, а формулирует вопросы, решение по которым принимает обучающийся.

```text
Ты — рецензент требований к системам искусственного интеллекта. Найди слабые места,
не исправляя их и не предлагая готовых формулировок. Для каждого дефекта укажи
идентификатор требования, тип дефекта и вопрос, на который автор должен ответить сам.

Типы дефектов:
1. НЕИЗМЕРИМО — нет способа проверить выполнение.
2. МЕТРИКА БЕЗ КОНТЕКСТА — значение метрики без baseline, распределения классов
   или условий замера.
3. ПРОТИВОРЕЧИЕ — два требования невыполнимы одновременно.
4. ПРОБЕЛ ИИ-СПЕЦИФИКИ — отсутствуют требования к данным, дрейфу, деградации,
   недетерминизму, объяснимости, справедливости, переобучению.
5. НЕТРАССИРУЕМО — требование не связано с бизнес-целью.
6. РЕШЕНИЕ ВМЕСТО ТРЕБОВАНИЯ — зафиксирована реализация вместо потребности.

Формат ответа: таблица | ID | Тип | Вопрос автору |.
В конце укажи отсутствующие разделы и три наиболее опасных дефекта с обоснованием.
```

Способ верификации: обучающийся указывает в декларации замечания, с которыми не согласился, с обоснованием.

## Формальный судья

Применяется в системе непрерывной интеграции. Проверяет только формальные признаки, не выставляет содержательную оценку и баллы.

```text
Ты — валидатор формы документа. Не оценивай качество содержания и не выставляй баллов.
Проверь документ и верни строгий JSON:

{
  "sections_present": {"<раздел>": true|false},
  "requirements_total": <число>,
  "requirements_without_id": [<цитаты>],
  "requirements_without_acceptance_criteria": [<ID>],
  "ai_disclosure_present": true|false,
  "ai_disclosure_has_unverified_section": true|false,
  "traceability_orphans": [<ID>],
  "verdict": "PASS" | "FAIL",
  "fail_reasons": [<строки>]
}

Правило вердикта: FAIL при отсутствии обязательного раздела, требований без ID
или декларации ИИ. Обязательные разделы задаются под конкретный КИМ.
Не добавляй ничего вне JSON.
```

Способ верификации: промпт калибруется на эталонном корректном и эталонном дефектном документах перед применением к потоку.

## Генератор дефектного артефакта

Применяется преподавателем для подготовки материала к инспекции. Запрет на явное указание неполноты обеспечивает естественность дефектов.

```text
Составь спецификацию требований на систему прогнозирования спроса для сети розничных
магазинов (интеграция с ERP и WMS, горизонт 14 дней, около 200 магазинов).
Оформи профессионально: структура по ISO/IEC/IEEE 29148, нумерованные требования,
приоритеты MoSCoW, раздел нефункциональных требований.

Ограничения: объём 6–8 страниц; не включай раздел о дрейфе данных; не включай
поведение при отказе модели; качество модели задай одним числом точности без
распределения классов и baseline; требования к данным опиши в общих словах;
этические аспекты упомяни одной фразой без метрики; не помечай документ как учебный.
```

Способ верификации: преподаватель составляет эталонный перечень дефектов до выдачи задания. Артефакт обновляется ежегодно.
