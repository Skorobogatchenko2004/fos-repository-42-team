# Методические указания преподавателям по использованию ресурсов

Указания определяют инструментальный контур дисциплины: какие инструменты применяются в каждом модуле, как они собираются в связки с оценочными средствами и по каким критериям выбираются. Статусы инструментов проверены 17.07.2026 по первоисточникам и подлежат ежегодной сверке (раздел 8).

## 1. Принцип отбора

Инструмент включается в дисциплину при условии, что артефакт ведётся в текстовом формате под контролем версий. Это условие обеспечивает работу модели оценивания: история коммитов показывает процесс, различия версий делают изменение предметом защиты, pull request превращает рецензирование в оценочное средство, непрерывная интеграция выполняет формальную проверку. При выборе между функционально удобным и текстовым инструментом предпочтение отдаётся текстовому.

## 2. Границы дисциплины

Разработка программного обеспечения с участием искусственного интеллекта охватывает семь этапов: сбор требований, проектирование, написание кода, тестирование, ревью, развёртывание, сопровождение. Дисциплина занимает этапы сбора требований и проектирования, частично — сопровождение. Написание кода, тестирование, ревью и развёртывание относятся к постреквизитам («Промышленная разработка систем ИИ», «MLOps»).

В связи с этим в дисциплину не включаются инструменты написания кода: средства автодополнения (GitHub Copilot, Tabnine), ИИ-среды разработки и агенты-кодеры (Cursor, Claude Code, Codex, Roo Code, Windsurf), платформы генерации приложений (Lovable, Base44), автономные агенты (Devin, OpenHands, AutoGPT). Исключение — проектная работа, где применение агента-кодера для прототипа допустимо при обосновании выбора (индикатор LC-1.2).

## 3. Обоснование дисциплины в условиях автономных агентов

Автономные агенты повышают, а не снижают цену нечётко поставленной задачи. При неясной постановке и высокой автономности исполнителя результат оказывается дорогостоящим, но архитектурно слабым: агент интерпретирует задачу самостоятельно и оптимизирует локально. При этом репозиторий выглядит рабочим, а процесс — продуктивным. Отсюда следует, что системная инженерия (чёткая постановка требований и архитектуры до передачи задачи исполнителю) сохраняет и повышает свою значимость.

Риски автономной разработки (недетерминированное поведение, циклическое выполнение, подмена цели прохождением проверок, расширение области изменений, рост стоимости, низкая прозрачность) используются как учебный материал модуля 4 при оценке рисков недетерминированного поведения моделей.

## 4. Инструменты по модулям

### Модуль 1

Feasibility Study и карта заинтересованных сторон ведутся в Markdown под контролем версий. Генеративные модели применяются в роли консультанта (формулировка задачи, разбор предметной области, подбор источников). Учебная задача модуля — проверка количественных оценок, которые модель порождает правдоподобно, но недостоверно.

### Модуль 2. Управление требованиями

| Инструмент | Статус | Лицензия | Роль |
|---|---|---|---|
| Markdown + Git + скрипт CI | — | — | основной контур |
| StrictDoc | 0.27.0 (2026) | Apache-2.0 | опционально: трассируемость, ReqIF |
| Doorstop | 3.2 (2026) | LGPL-3.0 | опционально: контроль подозрительных связей |

Основной контур — Markdown под контролем версий. Специализированные системы управления требованиями изучаются как отраслевой контекст, но в восьмичасовой лабораторной работе избыточны. Открытых редакторов формата ReqIF в рабочем состоянии нет; ReqIF рассматривается как формат обмена.

### Модуль 3. Архитектура

| Инструмент | Статус | Лицензия | Роль |
|---|---|---|---|
| PlantUML | 1.2026.6 | LGPL-3.0 | основной |
| C4-PlantUML | 2.13.0 | MIT | основной |
| MADR | 4.0.0 | MIT / CC0 | архитектурные решения |
| Structurizr (режим `local`) | 2026 | Apache-2.0 | опционально: просмотр модели |
| Mermaid | 11.16.0 | MIT | простые диаграммы, кроме C4 |

Диаграммы C4 разрабатываются в PlantUML. Диаграммы C4 в Mermaid имеют экспериментальный статус и не отображаются в GitHub нативно. Инструментальная обвязка архитектурных решений не требуется: ADR ведётся как Markdown-файл. Structurizr Lite использовать нельзя — репозиторий архивирован; применяется консолидированный образ в режиме `local`.

**Модельно-ориентированная системная инженерия.** Для занятия 3.3 рекомендуется лекционное изложение MBSE с практикой на C4-PlantUML. Причина — тяжёлые MBSE-среды (Capella, Papyrus) хранят модель в формате, который не сливается при командной работе. При необходимости практики применяется текстовый SysML v2 с графическим редактором Eclipse SysON.

Лабораторные работы модуля выполняются на фиксированных версиях с отключённой телеметрией. ## 5. Связки инструментов с оценочными средствами

- **Requirements-as-Code** ([КИМ-2.5](../../M2-requirements/kim-2.5-srs.md)): Markdown + Git → требования с идентификаторами → CI → критик-промпт → взаимное рецензирование → защита с анализом влияния изменения.
- **Architecture-as-Code** ([КИМ-3.4](../../M3-architecture/kim-3.4-architecture.md)): спецификация требований → C4-PlantUML → MADR → CI (рендеринг, трассируемость) → защита.
- **Quality-as-Code** ([КИМ-4.1](../../M4-quality-assurance/kim-4.1-vv.md)): DVC / MLflow → проверки данных и модели → CI → вердикт AI-PM.
- **Контур преподавателя**: GitHub Classroom → pull request → CI (формальная проверка, нулевой вес) → LLM-судья (только форма) → взаимное рецензирование → защита.

## 6. Архитектурные фитнес-функции

В модуле 3 архитектуру оценивает преподаватель; часть проверки может быть автоматизирована архитектурными фитнес-функциями — правилами, которым модель обязана удовлетворять, с выдачей результата в формате SARIF (отображается в pull request).

Работающая реализация — **aact** (GPL-3.0, парсер C4-PlantUML и Structurizr, встроенные и пользовательские правила, вывод SARIF). На архитектурах систем ИИ встроенные правила дают систематические ложные срабатывания: аксиома «у базы данных один владелец» противоречит намеренно разделяемым хранилищам ML-платформ (feature store, реестр моделей), а правило ацикличности отмечает как нарушение цикл переобучения. При прогоне архитектуры прогнозирования спроса с настройками по умолчанию все срабатывания ложные; ML-безопасная конфигурация оставляет пять правил из восьми и не выявляет специфичных для ИИ дефектов.

**Роль в дисциплине:**

- **демонстрация** (занятие 3.1 или 3.2) — основное применение. Иллюстрирует тезис о доменной специфичности фитнес-функций: правила микросервисного бэкенда не переносятся на системы ИИ. Настройка правил под предметную область является работой AI-PM;
- **разработка собственных правил** — тема повышенной сложности для проектной работы или дополнительного балла;
- в обязательную часть не включается: при безопасной конфигурации сигнал по предмету дисциплины близок к нулю, матрица трассируемости встроенными правилами не покрывается.

Проект развивается силами одной компании, опубликованных релизов нет, версия 3 находится в разработке; для учебного применения обязательна фиксация версии.

## 7. Критерии выбора инструмента

Отраслевая рамка выбора — последовательность «тип задачи → размер проекта → требования к данным → бюджет → уровень автономности» — дополняется критериями учебного применения:

| Критерий | Обоснование |
|---|---|
| Доступность из Российской Федерации | иначе задание неисполнимо на равных условиях |
| Бесплатность для обучающегося | платный инструмент в обязательном задании недопустим |
| Активность проекта | заброшенный инструмент делает задание неработоспособным |
| Порог входа | освоение не должно превышать по трудоёмкости само задание |

Критерии выбора применяются обучающимися при выборе инструмента для проекта и оцениваются по критерию «контроль над ИИ».

## 8. Минимальный вариант и регламент сверки

Дисциплина полностью обеспечивается четырьмя средствами: Git и GitHub Classroom; Markdown; PlantUML с C4-PlantUML; доступная генеративная модель. Остальные инструменты добавляются при необходимости.

Инструментальный ландшафт меняется быстрее, чем читается дисциплина. Список инструментов сверяется перед каждым семестром, дата сверки фиксируется в карточке ресурса, ответственный назначается в разделе [«Команда проекта»](../../team/README.md). Проверяются активность проекта (по дате релиза, а не последнего коммита), лицензия и работоспособность ссылок.
