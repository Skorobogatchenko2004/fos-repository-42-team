# Методические указания преподавателям по оцениванию

Указания определяют модель оценивания дисциплины в условиях, когда любой текстовый артефакт может быть подготовлен средствами генеративного искусственного интеллекта.

## 1. Исходное положение

Оценочные средства дисциплины — документы: Feasibility Study, техническое задание и спецификация требований, матрица трассируемости, архитектурное описание, программа и методика испытаний, план MLOps. Эти артефакты относятся к классу, который генеративные модели создают с высоким качеством. Задание вида «разработайте техническое задание» проверяет умение сформулировать запрос к модели, а не системное мышление.

В отличие от программного кода, документ не имеет ни автотестов, ни объективного вердикта о прохождении. Единственный барьер, который генеративная модель не преодолевает за обучающегося, — защита. Поэтому вес оценки смещается с артефакта на способность его защитить.

## 2. Принцип: ИИ является предметом оценки, а не нарушением

Запрет на использование искусственного интеллекта в дисциплине методически неверен по трём причинам:

- индикатор LC-1.2 требует выбора технологий под задачу; выбор инструмента ИИ является реализацией этого индикатора;
- целевые роли (AI-PM, ML Engineer) предполагают применение ИИ-инструментов в профессиональной деятельности;
- достоверное отличие сгенерированного текста от написанного вручную невозможно.

Использование ИИ разрешается на всех этапах и подлежит обязательному декларированию. Оценивается качество контроля над результатом.

## 3. Три контура проверки

### Контур 1. Формальный (допуск)

Проверяет наличие обязательных разделов, уникальность идентификаторов, отсутствие несвязанных элементов в матрице трассируемости, рендеринг диаграмм, заполненность декларации использования ИИ. Реализуется системой непрерывной интеграции и формальным LLM-судьёй. Вес — нулевой: работа, не прошедшая контур 1, к оцениванию не принимается.

LLM-судья применяется **только** для формальных признаков. Содержательную оценку он не выставляет: генеративные модели систематически завышают оценку связному тексту независимо от инженерной осмысленности.

### Контур 2. Экспертный (содержание)

Преподаватель и взаимное рецензирование. Проверяется обоснованность решений, работа со спецификой ИИ-систем, разрешение противоречий, качество trade-off анализа. Оцениваются индикаторы КРМ. Ориентировочный вес — около 40 %.

### Контур 3. Защита (понимание)

Устная защита по протоколу с вопросами по конкретным фрагментам работы и по истории изменений. Ориентировочный вес — около 60 %. Артефакт без защиты не оценивается.

## 4. Типы заданий, устойчивые к генерации

Задание устойчиво к генерации, если его результат зависит от ранее выполненной обучающимся работы либо требует вердикта, за который отвечает человек. В дисциплине применяются пять типов.

1. **Анализ влияния изменения.** Обучающийся проводит impact analysis по собственной матрице трассируемости после изменения условий. Результат зависит от качества ранее построенной матрицы.
2. **Инспекция дефектного сгенерированного артефакта.** Обучающийся проводит инспекцию заведомо дефектного документа, подготовленного средствами ИИ, и классифицирует дефекты.
3. **Вердикт по данным.** Обучающийся строит контур проверки готовой модели и выносит решение о вводе в эксплуатацию.
4. **Защита по истории коммитов.** Вопросы задаются по конкретным изменениям в репозитории.
5. **Взаимное рецензирование и деловые игры.** Одна команда защищает решение, другая проверяет его на устойчивость.

## 5. Декларация использования ИИ

Обязательный раздел каждой сдаваемой работы (форма приведена в [указаниях обучающимся](../students/README.md)). Отсутствие или недостоверность декларации — основание для возврата работы без оценки. Раздел «Не проверено» оценивается положительно: способность очертить границу проверенного является проверяемым результатом.

## 6. Общая рубрика

Единая шкала практических КИМ. Конкретные веса приведены в файлах КИМ.

| Критерий | Проверяемое содержание | Контур |
|---|---|:--:|
| Допуск | Формальная полнота, декларация ИИ | 1 |
| Специфика ИИ-систем | Требования к данным, дрейфу, деградации, недетерминизму, объяснимости | 2 |
| Обоснованность | Trade-off с явными альтернативами, связь решений с требованиями | 2 |
| Защита содержания | Обоснование решений, понимание последствий | 3 |
| Контроль над ИИ | Качество декларации, полнота описания непроверенного | 3 |

## 7. Материалы преподавателя

Раздел содержит закрытые материалы, не выдаваемые обучающимся:

- эталонные перечни дефектов сгенерированных артефактов для КИМ-2.5 и КИМ-3.4 с классификацией;
- описание заложенных дефектов модели и эталонное решение для КИМ-4.1;
- варианты вводных для анализа влияния изменения с картой ожидаемых затронутых требований;
- банки вопросов для защиты.

Дефектные артефакты и заложенные дефекты обновляются ежегодно. Ответственный за актуализацию назначается в разделе [«Команда проекта»](../../team/README.md).

## 8. Организационные условия

Для проведения оценивания необходимы: равный доступ обучающихся к генеративным моделям (вузовский шлюз либо разрешённые сервисы); система непрерывной интеграции для формальной проверки; согласование правила «артефакт без защиты не оценивается» с локальными нормативными актами.
