# КИМ-3.4. Моделирование системной архитектуры ИИ-системы

| Поле | Значение |
|---|---|
| Модуль | 3. Архитектура систем |
| Элемент дисциплины | Занятие 3.4 |
| Форма занятия | Практическое (лабораторная работа), 8 а.ч. и самостоятельная работа |
| Форма контроля | Текущий, защита лабораторной работы |
| Проверяемые индикаторы | LC-3.1 (уровень С); LC-1.2 (уровень С); ОПК-4.1 (уровень Б) |

## 1. Назначение

КИМ проверяет способность спроектировать архитектуру системы искусственного интеллекта, обосновать архитектурные решения и оценить последствия их изменения.

**Дескриптор освоения (LC-3.1, уровень С).** Обучающийся выбирает архитектурные паттерны под требования спецификации, обосновывает выбор через trade-off анализ с явными альтернативами и выделяет компоненты, обеспечивающие полный жизненный цикл модели машинного обучения.

**Дескриптор освоения (LC-1.2, уровень С).** Обучающийся проводит trade-off анализ и выбирает технологический стек под требования, разрешая противоречия между функциональными и нефункциональными ограничениями (часть Б, архитектурные решения).

**Дескриптор освоения (ОПК-4.1, уровень Б).** Обучающийся оформляет архитектурное описание с опорой на действующие стандарты: ISO/IEC/IEEE 42010:2022 (архитектурные представления) и ГОСТ Р 71539-2024 / ИСО-МЭК 5338-2023 (процессы жизненного цикла системы ИИ). Ссылка на устаревшую редакцию стандарта является дефектом оформления.

## 2. Предметная область

Система прогнозирования спроса для сети розничных магазинов — сквозная предметная область дисциплины (см. [КИМ-2.5](../M2-requirements/kim-2.5-srs.md), раздел 2).

## 3. Условия выполнения

Работа выполняется индивидуально в личном репозитории (GitHub Classroom), срок — 2 недели. Архитектурное описание ведётся в текстовом формате под контролем версий: диаграммы в текстовой нотации, архитектурные решения — отдельными файлами.

Использование генеративного искусственного интеллекта разрешено и подлежит обязательному декларированию. Работа без заполненной декларации к проверке не принимается.

## 4. Задание

### Часть А. Архитектурное описание (C4, уровни 1–3)

Разработать средствами PlantUML с библиотекой C4-PlantUML:

1. диаграмму контекста (C4 уровень 1) — система, внешние акторы, смежные системы (ERP, WMS, BI);
2. диаграмму контейнеров (C4 уровень 2) с обязательным выделением компонентов, специфичных для системы ИИ: хранилище и подготовка признаков, обучающий конвейер, реестр моделей, сервис инференса, конвейер мониторинга качества данных и дрейфа, контур переобучения с триггерами, контур human-in-the-loop (при наличии в требованиях);
3. диаграмму компонентов (C4 уровень 3) для одного контейнера по выбору обучающегося с обоснованием выбора;
4. диаграмму последовательности для сценария ежедневного пересчёта прогноза;
5. диаграмму потоков данных с указанием точек контроля качества данных;
6. диаграмму развёртывания.

Каждая диаграмма сопровождается таблицей трассируемости «элемент архитектуры → идентификаторы требований». Элемент, не связанный ни с одним требованием, удаляется либо оформляется запросом на изменение спецификации.

### Часть Б. Архитектурные решения

Оформить три архитектурных решения по шаблону MADR. Обязательные темы: способ доставки прогноза потребителям; стратегия обновления модели; поведение системы при недоступности или низкой уверенности модели. Каждое решение содержит контекст, не менее двух рассмотренных альтернатив, критерии сравнения, решение и последствия, включая негативные.

### Часть В. Инспекция сгенерированной архитектуры

Провести инспекцию архитектурного описания, подготовленного средствами генеративного искусственного интеллекта (выдаётся преподавателем). Выявить не менее пяти существенных инженерных дефектов. Для каждого дефекта указать нарушенное требование, последствия в эксплуатации и способ исправления.

### Часть Г. Анализ влияния изменения

Вводная выдаётся на защите: развёртывание только on-premise, данные не покидают контур предприятия, доступные GPU отсутствуют. Устно, с опорой на собственную архитектуру, определить затронутые контейнеры, аннулированные архитектурные решения, ставшие невыполнимыми требования и предложить альтернативу.

## 6. Критерии и шкала оценивания

Максимальная сумма — 100 баллов. Работа, не прошедшая формальную проверку, к оцениванию не принимается.

| № | Критерий | Проверяемое содержание | Баллы |
|:--:|---|---|:--:|
| 0 | Допуск | Формальная проверка пройдена, декларация ИИ заполнена | не оценивается / не принято |
| 1 | Полнота описания | Шесть диаграмм, выделены компоненты, специфичные для ИИ | 10 |
| 2 | Трассируемость | Каждый элемент связан с требованием, несвязанных элементов нет | 15 |
| 3 | Качество архитектурных решений | Реальные альтернативы, явные критерии, названы негативные последствия | 15 |
| 4 | Инспекция | Найденные дефекты, корректность классификации, связь с требованиями | 15 |
| 5 | Защита содержания | Обоснование решений, понимание их последствий | 25 |
| 6 | Анализ влияния | Определение затронутых элементов, аннулированных решений, альтернатива | 10 |
| 7 | Контроль над ИИ | Качество декларации, полнота описания непроверенного | 10 |

Шкала перевода: 85–100 — отлично; 70–84 — хорошо; 55–69 — удовлетворительно; менее 55 — не зачтено.

Работа, набравшая менее 10 баллов из 25 по критерию 5, не засчитывается независимо от суммы.

**Ориентир для критерия 3.** Архитектурное решение считается обоснованным, если названы негативные последствия выбранного варианта и указана его цена. Решение с формально приведёнными альтернативами и без явных негативных последствий признаётся незаполненным.

## 7. Правила использования генеративного ИИ и внешних ресурсов

Разрешается использование генеративных моделей для подготовки черновиков диаграмм, архитектурных решений и текста, а также поисковых сервисов и открытых источников.

Обязательно: заполнение декларации использования ИИ; ведение осмысленной истории коммитов; проверка ссылок на стандарты и паттерны по первоисточнику. Ссылка на несуществующий стандарт снимает баллы по критерию 7 полностью.

Не допускается сдача сгенерированного текста без проверки.
