# Карточка требования (шаблон)

Каждое требование оформляется по приведённой структуре. Идентификатор стабилен и не переиспользуется после удаления требования.

```markdown
### REQ-000. Краткое наименование

- **Тип:** функциональное / нефункциональное (качество модели, данные, производительность, безопасность, этика)
- **Приоритет:** Must / Should / Could / Won't (MoSCoW)
- **Источник:** идентификатор бизнес-цели (BG-00) или заинтересованной стороны
- **Формулировка:** проверяемое утверждение о поведении или свойстве системы
- **Критерий приёмки:** способ и условия проверки выполнения
- **Обоснование значения:** для числовых порогов — обоснование выбранного значения
```

## Пример заполнения

```markdown
### REQ-014. Обнаружение дрейфа распределения продаж

- **Тип:** нефункциональное / качество модели
- **Приоритет:** Must
- **Источник:** BG-02 (снижение потерь от неликвида)
- **Формулировка:** Система еженедельно рассчитывает индекс PSI между обучающей
  выборкой и продажами за последние 7 дней по каждой товарной категории.
- **Критерий приёмки:** при PSI > 0,2 формируется оповещение менеджеру категории
  в течение 1 часа; факт оповещения фиксируется в журнале.
- **Обоснование значения:** порог 0,2 соответствует отраслевой практике для
  розницы и подлежит калибровке по итогам опытной эксплуатации.
```

## Правила

- формулировка описывает потребность, а не реализацию (не «использовать XGBoost», а требуемое свойство);
- каждое требование связано с бизнес-целью в матрице трассируемости;
- числовой порог без обоснования значения считается неполным.
